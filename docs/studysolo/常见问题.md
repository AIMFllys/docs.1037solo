# 常见问题 (FAQ)

### Q1: 为什么有时系统规划出来的学习节点图有环路的连线，或者点“运行”直接报出拦截信息？
**A1:** StudySolo 的工作流引擎原理基石在于数据拓扑排序（Topology Sort）。如果手动连接画布时不小心将产出端又接回了最上层的源头形成了“无限死循环的环闭路线”，算力流注将会永远被锁死，所以引擎启动防爆机制抛错了。只需剪断形成闭环的连线即可恢复引擎传导。

### Q2: 画布上面的防抖存档提示是什么意思？我会因为刷新丢弃所有流程图数据吗？
**A2:** 为了让你在海量节点的屏幕上拖拽操作依然享受到 120 帧极致流畅体验，我们未将庞大坐标变动实时发起无意义密集保存，而是应用了业界顶尖的“三级泄洪本地缓存”——所有的更改会被先行储存在浏览器核心的底层容器 (IndexedDB) 中。系统侦测你停止拖沓超 过 4 秒钟，才会悄然在冰山底端与云端 Supabase 完成状态锁定。即便是遇到强行关机，本地 IndexedDB 中的备用帧快照也极大程度会找回状态。

### Q3: 这边的登录资产机制能独立充值扩容吗，还是一定要绕回主项目充值？
**A3:** 因为整个体系搭载由 `1037Solo` 的 Supabase 发出的无缝穿梭单点通行证，您于本产品或主产品的配额、订阅层级，积分令牌都是全生命周期通用的。资产可以在两界内随意调度、消耗或充值，它只认同你在这个宇宙下的 One-ID。

### Q4: 个别执行完的生成的字块变成了刺眼的红色警告块是为什么？
**A4:** 那代表对应的独立代理进程执行遇到了不可挽回の阻塞（这可能是上游模型因安全政策拒绝应答，或者传输过程中解析 JSON 发生粉碎性错位）。我们的节点不会让这种恶劣状态默默向全局污染，而是自动把断口隔离熔断成红色提醒（Failed Node），并在节点内打印出导致报错的内部堆栈回溯（StackTrace），只需修改上游源头或调整相关指示语句然后对该单一卡点单独特权“重跑此节点”即可冲破封锁！
